=> Authentication vs Authorization
    -> Authentication means verify identity
    -> Authorization means what you are allowed to do.

=> What is the Microsoft Identity Platform?
    -> The Microsoft identity platform is a system provided by Microsoft that allows applications to authenticate users and get authorized access to resources and APIs.
    -> The identity platform supports multiple types of users.
        -> Microsoft accounts: These are personal accounts like: Outlook, Xbox, Hotmail
        -> Work or school accounts: These are organizational accounts created in: Microsoft Entra ID. These accounts are used inside organizations.
        -> Social or Local Accounts Using B2C: Applications that serve customers can use: Azure Active Directory B2C
            -> B2C means Business to Consumer
            -> It allows login using: Google, Facebook, Local accounts, Microsoft accounts
            -> It is used when your application is used by public customers.
            -> Example customers: Shopping website users, Banking app users, Mobile app users, Online service users. These users are not employees of your company.
        -> External Customer Accounts(Microsoft Entra External ID)
            -> This allows organizations to authenticate: Customers, Partners, External users
            -> Example: Partner portals or B2B applications.
            -> This is used when external people need access to your organization's resources.
            -> These people are not employees, but they collaborate with your company.
            -> Example external users: Partners, Suppliers, Vendors, Contractors, Customers accessing internal portals
            -> These users are not employees, but they need secure access. So the company uses Microsoft Entra External ID.
    -> Developer does not need to create a custom authentication system. Applications can allow users to log in using existing identities.
    -> The identity platform handles the authentication process.
    -> Components of the Microsoft Identity Platform: There are several component of MIP
            -> Authentication Service (Standards-based)
                -> Authenticate users using OAuth / OpenID
                -> It supports protocols like:
                    -> OAuth 2.0: OAuth 2.0 is a standard authorization framework. It allows applications to access resources on behalf of a user.
                    -> OpenID Connect: OpenID Connect is an authentication layer built on top of OAuth 2.0. It allows applications to verify user identity.
                -> This service allows developers to authenticate multiple identity types such as:
                    -> Work or school accounts from Microsoft Entra ID
                    -> Personal Microsoft accounts (Outlook, Xbox)
                    -> Social accounts through Azure Active Directory B2C
                    -> Customer identities through Microsoft Entra External ID
                -> So this component is responsible for: User authentication, Identity verification and Token issuance
            -> Authentication Libraries(Microsoft Authentication Library)
                -> Help developers integrate authentication
                -> These libraries help developers: Sign in users, Acquire tokens, Call APIs securely
                -> Supported platforms include: Java, Javascript, Python, .NET, Android, iOS
            -> Identity Platform Endpoint
                -> API endpoint for authentication and tokens
                -> Example endpoint: https://login.microsoftonline.com
                -> Applications send requests to this endpoint for: User authentication, Token requests, Authorization
                -> The endpoint works with: MSAL, Any OAuth-compliant library
            -> Application Management Portal
                -> Register and configure applications
                -> Applications must be registered in Azure. This is done through: Azure Portal → App Registrations
                -> Here developers configure:
                    -> Application ID
                    -> Redirect URI
                    -> Permissions (Scopes)
                    -> Secrets / Certificates
                -> This configuration allows the identity platform to recognize the application.
            -> Application Configuration API
                -> Automate application configuration
                -> This is fifth component allows programmatic management of applications.
                -> Developers can configure applications using:
                    -> Microsoft Graph
                    -> Microsoft PowerShell
                -> This helps in:
                    -> DevOps automation
                    -> CI/CD pipelines
                    -> Infrastructure automation
    -> Example: Suppose you build an e-commerce website. Users should be able to login using: Email/password, Google, Facebook
       Instead of implementing Facebook login yourself, you integrate your application with:
           -> Microsoft identity platform
           -> Azure Active Directory B2C (used for customer identities)
       Now users can login with Facebook.
       -> High Level Architecture
            User
              ↓
            Your Web Application
              ↓
            Microsoft Identity Platform (Azure AD B2C)
              ↓
            Facebook Identity Provider
              ↓
            Token returned
              ↓
            User logged into your application
       -> Step-by-Step Login Flow (Facebook Login)
            -> User clicks "Login"
                -> Your application does not perform authentication itself.
                -> Instead, it redirects the user to an Identity Platform (Identity Provider).
                -> Example: Microsoft identity platform.

            -> Request Sent to Identity Platform
                -> Your application sends an authentication request to the Identity Platform.
                -> Example request parameters:
                    -> client_id = your_app_id
                    -> scope = openid profile email
                    -> redirect_uri = your_app_url
                    -> response_type = code
                -> This request follows OAuth 2.0 and OpenID Connect standards.

            -> Identity Platform Processes the Request
                -> The Identity Platform checks the application configuration.
                -> It verifies which identity providers are configured for login.

            -> Identity Platform Displays Login Page
                -> The login page is generated by the Identity Platform.
                -> The user is presented with multiple login options.

            -> Login Options May Include
                -> Social login providers
                    -> Facebook
                    -> Google
                    -> Apple
                -> Direct sign-in using credentials
                    -> Email / Username
                    -> Password
                -> This is usually called Local Account authentication.

            -> User Selects Facebook Login
                -> The Identity Platform redirects the user to Facebook for authentication.
                -> Facebook asks the user:
                    -> "Do you allow this application to access your profile?"

            -> User Grants Permission
                -> The user clicks "Allow".

            -> Facebook Authenticates the User
                -> Facebook verifies:
                    -> User credentials
                    -> Valid Facebook account

            -> Facebook Generates Authentication Result
                -> After successful authentication, Facebook generates a token confirming the user's identity.

            -> Facebook Sends Result Back to Identity Platform
                -> The authentication result is returned to the Identity Platform.
                -> Example Identity Platform: Microsoft identity platform.

            -> Identity Platform Generates Authorization Code
                -> Instead of sending tokens directly to the application, the Identity Platform generates an Authorization Code.

            -> Redirect Back to Your Application
                -> The Identity Platform redirects the user back to your application.
                -> The Authorization Code is sent to the redirect_uri.

            -> Application Sends Authorization Code to Identity Platform
                -> Your application sends the Authorization Code to the Identity Platform's token endpoint.
                -> This request includes:
                    -> client_id
                    -> client_secret
                    -> authorization_code
                    -> redirect_uri

            -> Identity Platform Verifies the Authorization Code
                -> The Identity Platform verifies the authorization code and application identity.

            -> Identity Platform Issues Tokens
                -> After successful verification, the Identity Platform returns:
                    -> ID Token (contains user identity information)
                    -> Access Token (used to call APIs)
                    -> Refresh Token (optional, used to obtain new access tokens)

            -> Application Verifies the ID Token
                -> Your application validates the ID Token to confirm the user's identity.

            -> User Logged Into Application
                -> Once the token is verified, the application trusts the identity.
                -> The user is successfully logged into the system.

            -> Application Uses Access Token
                -> The Access Token is used to call protected APIs such as:
                    -> Microsoft Graph
                    -> Custom backend APIs

    -> Microsoft Graph is an API gateway that allows applications to access Microsoft services like:
        -> Outlook
        -> Teams
        -> OneDrive
        -> Calendar
        -> Users in Entra ID
    -> Example: Your app may call Graph API to: Read user profile, Send email, Get calendar events. But the user must authorize access first.

=> What is Microsoft Entra ID (Azure Active Directory)

    -> Simple Real-World Example

=> App Registration
    -> An App Registration represents an application definition in Microsoft Entra ID.
    -> When you register an application, you are telling Azure: “This application exists and may request authentication.”
    -> It creates a global identity for the application.
    -> Key characteristics:
        -> Represents the application itself
        -> Used when apps need to authenticate users or services
        -> Can be used across multiple tenants (if configured)
    -> App Registration mainly stores:
        -> Application ID (Client ID)
        -> Authentication configuration(Redirect URI)
        -> API permissions

=> Why Application Registration Is Required
    -> Before an application can authenticate users or access APIs, it must be registered in Microsoft Entra ID.
    -> When you register an app, Azure creates an identity configuration for that application.
    -> This configuration allows the application to:
        -> Authenticate users
        -> Request tokens
        -> Access APIs
        -> Integrate with Azure identity system
    -> Example: Your Web App → Register in Entra ID → Gets identity
    -> Just like users have identities, applications also need identities.

=> Single-Tenant vs Multi-Tenant Applications
    -> When registering an application, you choose its accessibility.
    -> In Single Tenant Application, Application is accessible only inside your organization tenant.
        -> Example: Internal HR portal, Internal company dashboard: Only users from your tenant can log in.
    -> In Multi-Tenant Application, Application can be used by users from other organizations.
        -> Example: SaaS application, Microsoft Teams integrations, Third-party enterprise apps. Users from multiple tenants can authenticate.

=> What Happens When You Register an App
    -> When you register an application in the Azure portal, two objects are created automatically.
        -> Application Object(Global Definition of the App)
            -> It is a blueprint or template created in the tenant where the application was registered (called the home tenant).
            -> Properties stored in the application object include:
                -> Application ID (Client ID)
                -> Redirect URIs
                -> API permissions
                -> Token configuration
                -> Required scopes
            -> The schema for this object is defined by Microsoft Graph. The documentation gives a class analogy.
                Application Object = Class/Blueprint
                Service Principal = Object Instance in tenants
            -> The application object defines three important aspects of the application.
                -> Token Issuance: It defines how tokens are issued for accessing the application.
                    -> Example: OAuth configuration and OpenID Connect settings
                    -> Protocols involved: OAuth 2.0 and OpenID Connect
                -> Resources the App Needs:
                    -> The application object defines which resources the application wants to access.
                    -> Example: Microsoft Graph API, Custom backend APIs
                -> Actions the App Can Perform
                    -> It defines what actions the application is allowed to perform.
                    -> Example: Read user profile, Send email, Access calendar
        -> Service Principal Object (Identity of the App Inside a Tenant)
            -> To access resources in a tenant, there must be a security principal. Examples of security principals:
                -> User Principal → Represents a user
                -> Service Principal → Represents an application
            -> So a Service Principal is the identity of an application inside a tenant.
            -> Example: Suppose you build an application "MyApp". You register it in Tenant A
            -> Azure creates:
                -> Application Object → Global definition
                -> Service Principal → Identity of the app in Tenant A
            -> In case of Multi-Tenants, suppose another company uses your app.
                Tenant A → Original tenant.
                Tenant B → External organization
                Tenant C → Another organization
            -> When users from those tenants use your app:
                Application Object → Still one (global)
                Service Principals → Created in each tenant
            -> Types of Service Principals Object: Azure defines three types.
                -> Application Service Principal(manual): This is the most common type.
                -> Managed Identity Service Principal(automatic)
                -> Legacy Service Principal

                -> Application Service Principal
                    -> A Service Principal is the security identity of an application inside a tenant.
                    -> This is created when you:
                        -> Register an application in Azure AD(for example: If you build a custom app and registered it). It represents that app in your tenant.
                        -> Create a service principal manually
                        -> Provide credentials
                    -> Service Principal uses: Client ID (Application ID), Client Secret / Certificate and Assigned Roles / Permissions
                    -> Flow:
                        OpenShift / Kubernetes
                                ↓
                        Uses Service Principal Credentials
                        (Client ID + Client Secret)
                                ↓
                        Authenticate with Azure
                                ↓
                        Azure validates identity
                                ↓
                        Checks Role Assignment
                                ↓
                        ACR allows Image Pull
                    -> Think of it as the instance of the application inside a tenant.
                    -> Relationship:
                        App Registration (Application Definition)
                                │
                                ▼
                        Service Principal (Security Identity in Tenant)
                    -> Suppose your application wants to access: storage, container registry, database
                    -> You create a Service Principal and assign roles.
                        -> Example roles: Contributor, Reader, AcrPull, Storage Blob Data Reader
                    -> Now the application can authenticate using: Client ID and Client Secret

                -> Managed Identity Service Principal:
	                -> A Managed Identity is a service principal in Microsoft Entra ID that is fully managed by Azure.
                    -> It allows the resource to authenticate securely to other Azure services without storing credentials such as usernames or passwords.
                    -> Managed identities are issued and authenticated through Azure AD(Microsoft Entra ID).
                    -> It is fully managed by Azure, meaning Azure handles creation, lifecycle, and credential rotation.

                    -> Types of Managed Identity: There are two types.
                        -> System Assigned Managed Identity → Auto created + Auto linked to the resource
                            -> A System-assigned Managed Identity is directly associated with a specific Azure resource.
                            -> When this identity is enabled for a resource, Azure automatically creates an identity for that resource in Azure AD(Microsoft Entra ID) and provide you object id of that SAMI.
                            -> This identity is tightly linked to the lifecycle of the particular  resource.
                            -> Steps to enable system assigned Managed identity in a particular resource:
                                -> Go to Azure Portal and Create a container registery resource or any other like VM, App Service etc
                                -> Click on resource overview
                                -> In Left hand side Setting--->identity
                                -> Under System assigned managed identity, switch the status to on.
                                -> Click save
                                -> You will see Object(principal) ID below Status
                                -> Object(principal) ID:
                                    -> This is object id of Managed identity service principal created inside Azure AD
                                    -> After enabling System assigned, Azure automatically create a Managed identity service principal in you Azure AD tenant.
                            -> So, SAMI(System assigned Managed Identity) will be created inside the resource by simply toggling "On" under identity.
                            -> Key Characteristics
                                -> Created automatically when enabled on a resource.
                                -> Tied to one Azure resource only.
                                -> Cannot be shared with other resources.
                                -> Automatically deleted when the resource is deleted.
                                -> Simplest identity option for most use cases.
                            -> How It Works
                                -> When a resource such as a virtual machine or Kubernetes cluster needs to access another Azure service-
                                -> Azure uses its system-assigned identity to authenticate with Microsoft Entra ID.
                                -> If the identity has the required role permissions, access is granted.
                                -> Example: Suppose an application running in Azure Kubernetes Service needs to pull container images from Azure Container Registry.
                                -> Step:
                                    -> Enable system-assigned identity on the Kubernetes cluster.
                                    -> Assign the AcrPull role to that identity on the container registry.
                                    -> The cluster can now securely pull images without storing credentials.
                                -> Service like AKS, Azure VM commonly use system-assigned identities.

                        -> User Assigned Managed Identity: Manually created + Manually attached to the resource
                            -> A User-assigned Managed Identity is created as a separate Azure resource(like creating a VM or Storage account).
                            -> Once created, you can then attach/integrate that UAMI into one or more resources.
                            -> Unlike system-assigned identities, this identity is not tied to a single resource and remains available even if a resource using it is deleted.
                            -> Key Characteristics
                                -> Created as a separate Azure resource.
                                -> Can be shared across multiple services.
                                -> Lifecycle is independent of the resources using it.
                                -> Provides more flexibility in identity management.
                            -> How It Works
                                -> First, a user-assigned identity is created in Azure.
                                -> After that, this identity can be attached to different services such as virtual machines, app services, or container services.
                                -> When those services need to access another Azure service, they authenticate using this shared identity through Microsoft Entra ID.
                                -> Example: A company may have multiple services that need to access a centralized secrets store like: Azure Key Vault
                                    -> Instead of creating separate identities for each service, a single user-assigned identity can be created and attached to multiple applications. This identity is granted access to the Key Vault.
                                    -> This approach simplifies permission management.
                    -> Both types serve the same purpose: secure authentication between Azure services, but they differ in lifecycle and reuse capability.
                    -> Both identities rely on Microsoft Entra ID for authentication and integrate with Azure Role-Based Access Control (RBAC) to control permissions.
                    -> System-assigned identities are created automatically for a resource, whereas user-assigned identities are created manually but still managed by Azure.
                    -> Important Point:
                        -> A resource can have both SAMI and UAMI like SAMI, UAMI-1, UAMI-2..etc.
                        -> You can enable SAMI and also link UAMI on the same resource if you want.
                        -> During token request you can choose which identity to use.
                        -> Most Azure resources allow you to pick the identity when requesting a token.
                        -> Example for a VM: You call metadata endpoint:
                            -> Without identity ID -> Azure uses SAMI
                            -> With identity ID -> Azure uses UAMI
                        -> You must assign roles separately to each identity. SAMI & UAMI both have its own permissions.
                        -> Common scenario:
                            -> SAMI: Used for resource's own operations.
                            -> UAMI: Used for application logic(Because UAMI is reusable and portable)
                    -> How Authentication Works:
                        -> When the Azure service wants to access another Azure service: Example - AKS → Access → Azure Container Registry
                        -> The process is:
                            AKS
                             │
                             │ Uses Managed Identity
                             │
                             ▼
                            Microsoft Entra ID
                             │
                             │ Verifies identity (Authentication)
                             │
                             ▼
                            Access decision
                        -> For authorization: After authentication, Azure checks permissions using RBAC.
                            -> Example service: Azure Container Registry
                            -> Permission example: Role: AcrPull
                            -> Flow:
                                1️⃣ AKS proves identity → Authentication
                                2️⃣ Azure checks role → Authorization
                                3️⃣ Access granted or denied
                -> Mental Model:
                    Service Principal
                         │
                         ├── Application Service Principal (Manual)
                         │        Created by developer
                         │        Uses Client Secret / Certificate
                         │
                         └── Managed Identity Service Principal (Automatic)
                                  Created by Azure
                                  No credentials required

                -> Legacy Service Principal:
                    -> A Legacy Service Principal is an older type of service principal that was created before the modern App Registration model existed.
                    -> In the early days of Azure, applications could create service principals directly, without creating an application object (App Registration) first. So the identity looked like this:
                        Application
                              ↓
                        Legacy Service Principal
                    -> There was no App Registration / Application Object involved.

=> Step to Enable Authentication/Authorization using Managed Identity (Microsoft Entra ID)
    -> Goal:
        -> Allow Azure resource to securely access another Azure resource
        -> Avoid storing credentials (username/password, client secret)
    -> Steps:
    -> STEP 1 → Create User Assigned Managed Identity (Optional but Recommended)
    -> STEP 2 → Enable Managed Identity on Resource
        -> Enable SAMI or link UAMI created in step-1&2 (for more details about this refer Managed Identity Service Principal above)
    -> STEP 3 → Assign RBAC Role to Managed Identity
        -> To assign the role:
            -> Go to Target Resource (the resource you want to access). Example: Azure Container Registry / Blob Storage
            -> Open Access Control (IAM)
            -> Click Add → Add role assignment
    -> STEP 4 → Choose Role (Authorization). Select appropriate role based on requirement:
                    | Role        | Permission                |
                    | ----------- | ------------------------- |
                    | Owner       | Full control              |
                    | Contributor | Create & manage resources |
                    | Reader      | Only view resources       |
                    | AcrPull     | Pull images from ACR      |
    -> STEP 5 → Assign Access to Managed Identity
        -> In "Assign access to" → Select → Managed Identity
        -> Click +Select Members. You will see 2 options:
            -> System assigned -> Select your resource SAMI
            -> User Assigned -> Select your resource UAMI
            -> You can assign r0le to both or one or each one with different role.
    -> STEP 6 → Finalize Role Assignment
        -> Select required identity (SAMI / UAMI / both)
        -> Click Review + Assign
    -> STEP 7 → How It Works Internally
        → Resource (VM / AKS / App Service)
        		→ Uses Managed Identity
        			→ Authenticates with Microsoft Entra ID
        				→ Identity validated (Authentication)
        					→ RBAC checks role permissions (Authorization)
        						→ Access granted to target resource
    -> Final Summary:
        -> Managed Identity removes need for credentials
        -> SAMI → Auto created and linked to resource
        -> UAMI → Manually created and reusable
        -> RBAC controls what the identity can access
        -> Authentication handled by Microsoft Entra ID

=> Problem Before Managed Identity and how it is resolved
    -> Earlier when an application needed to access another Azure service, we used credentials.
    -> Example: App needs to access database. So we store username and password inside configuration file, environment variables, Kubernetes secret
    -> Problem: ❌ Security risk, ❌ Password rotation required, ❌ Credentials can leak
    -> To solve this problem, Azure introduced Managed Identity.:
        -> Managed Identity is an automatically managed identity for an Azure resource that allows it to authenticate to other Azure services without storing credentials.
        -> Important point: Managed Identity internally creates a Service Principal in Microsoft Entra ID. So technically
            Managed Identity
                  ↓
            Service Principal (automatically created)
        -> Traditional Service Principal (Manual Identity): When developers created identities manually, the architecture looked like this:
                Application
                     ↓
                App Registration
                     ↓
                Application Object
                     ↓
                Service Principal
                     ↓
                Authentication using Client ID + Client Secret
            -> Managed Identity Architecture: With Managed Identity, Azure automatically creates and manages the identity.
                Azure Resource (AKS / VM / App Service)
                            ↓
                Managed Identity Enabled
                            ↓
                Service Principal Automatically Created
                            ↓
                Authentication through Microsoft Entra ID
                            ↓
                Access Azure Resource
        -> So instead of using password: App proves its identity automatically
    -> Workflow:
        -> Without Managed Identity:
            AKS
              │
              │ Client ID + Secret / Username + Password(Problem: secret stored in config)
              ▼
            Microsoft Entra ID
              │
              │ Authentication
              ▼
            RBAC Permission Check
              ▼
            ACR Access
        -> With Managed Identity:
            AKS
              │
              │ Managed Identity
              ▼
            Microsoft Entra ID
              │
              │ Authentication
              ▼
            Azure RBAC Permission Check
              ▼

=> Why Permissions Are Needed
    -> When an application integrates with the identity system, it often needs to access another resource or API.
    -> Example resources: Microsoft Graph, Microsoft 365 APIs, Azure Key Vault (Your App → wants to read user calendar)
    -> But Azure cannot allow this automatically. Instead, permissions must be requested and approved. This model ensures:
        -> User control, Admin control, Security, Limited access
    -> Authorization Protocol Used: The identity platform uses the protocol OAuth2.O

=> What Are Scopes (Permissions)
    -> Resources expose permissions called scopes. A scope represents a specific action that an application can perform.
        -> Example resource: Microsoft Graph API
            -> Example permission: https://graph.microsoft.com/Calendars.Read
            -> Meaning: Application can read user calendars
        -> Example resource: Microsoft Graph API
            -> Example permission: https://graph.microsoft.com/Mail.Send
            -> Meaning: Application can send email as user
    -> So scopes break functionality into small permission units.

=> Resource Identifier (Application ID URI)
    -> Every resource has a unique identifier. Example
        Microsoft Graph → https://graph.microsoft.com
        Microsoft Mail API → https://outlook.office.com
        Azure Key Vault → https://vault.azure.net
    -> Permission format: resource_identifier / permission
    -> Example: https://graph.microsoft.com/Calendars.Read

=> How Apps Request Permissions
    -> Applications request permissions using the scope parameter in OAuth requests.
    -> Example request: GET https://login.microsoftonline.com/{tenant}/oauth2/v2.0/authorize
            -> Important parameters: client_id, redirect_uri, response_type, scope
            -> scope=https://graph.microsoft.com/Calendars.Read, https://graph.microsoft.com/Mail.Send
        -> Meaning: App requests permission to: Read calendar, Send mail

=> Permission Types: The identity platform supports two types of permissions.
    -> Delegated Permissions: Delegated permissions are used when: User is signed in. The application acts on behalf of the user.
        -> Example: User logs into an app-->App calls Microsoft Graph--> App reads user's calendar
    -> Application Permissions (App-Only)
        -> These are used when: No user is present
        -> Example: Background services, Scheduled jobs, Daemons, Server-to-server apps
        -> Example scenario: Nightly service → scans organization mailboxes. In this App accesses API directly.Only administrators can grant these permissions.

=> Consent Concept
    -> Before an application can use permissions, someone must approve them. This approval is called consent.
    -> Consent ensures:
        -> Users know what data the app can access
        -> Admins control organization data
    -> Types of Consent: There are three consent models.
        -> Static User Consent: Permissions are defined during App Registration in the Azure portal. Example: User.read, Mail.send, Calendars.read
            -> When the user signs in: App requests all permissions immediately
            -> User sees a consent screen like:
                App wants to:
                Read your calendar
                Send mail
                Read your profile
            -> Problems with static consent:
                Too many permissions shown
                Users may refuse access
                Hard to know future needs
        -> Incremental / Dynamic Consent: Instead of asking everything upfront, the app requests permissions only when needed.
            -> Example:
                User logs in → ask User.Read
                Later → ask Calendars.Read
                Later → ask Mail.Send
            -> Flow:
                User uses new feature
                      ↓
                App requests new permission
                      ↓
                User consents
            -> Dynamic consent works only for delegated permissions, not app-only permissions.
        -> Admin Consent: Some permissions are high-privilege.
            -> Example:
                Read all users in organization
                Read all mailboxes
                Access organization data
            -> These require administrator approval. Admins can approve on behalf of the entire organization.

=> Complete Permission Flow (End-to-End)
    Application
          ↓
    Redirect user to Microsoft identity platform
          ↓
    User signs in
          ↓
    App requests scopes
          ↓
    User/Admin gives consent
          ↓
    Access token issued
          ↓
    App calls API

=> Conditional Access in Microsoft Entra ID
    -> Conditional Access is a security feature in Microsoft Entra ID.
    -> It allows organizations to control how users access applications and services.
    -> Access is granted only when specific security conditions are satisfied.
    -> It is commonly used to enforce identity security policies before issuing tokens.

    -> Purpose of Conditional Access
        -> Protect applications and APIs
        -> Enforce additional security verification
        -> Reduce unauthorized access risks
        -> Apply security policies dynamically

    -> Security Controls Supported by Conditional Access
        -> Multifactor Authentication (MFA)
        -> Allow access only from Intune enrolled devices
        -> Restrict access based on location
        -> Restrict access based on IP ranges
        -> Require compliant or managed devices

    -> How Conditional Access Impacts an Application
        -> In most cases Conditional Access does NOT require application code changes.
        -> The authentication flow automatically handles the security policy.
        -> However, some scenarios require the application to handle Conditional Access challenges.

    -> Scenarios That May Require Code Changes
        -> Applications performing On-Behalf-Of (OBO) flow
        -> Applications accessing multiple APIs or resources
        -> Single Page Applications using MSAL.js
        -> Web applications calling a protected API

    -> Why Conditional Access Challenges Occur
        -> When the requested resource has a security policy enabled
        -> When additional verification like MFA is required
        -> When a downstream API requires stronger authentication

    -> Conditional Access Challenge Flow
        User Sign-In
        		→ Application requests access token
        			→ Identity platform checks Conditional Access policies
        				→ If policy requirement exists
        					→ Conditional Access challenge returned
        						→ Application prompts user for additional authentication
        							→ Token issued after successful verification

    -> Example Scenario 1 (No Code Change Required)
        -> Scenario: A single-tenant mobile application is developed. The application signs in the user but does not call any API.
        -> Process:
            → User signs in
            → Conditional Access policy triggers MFA
            → User completes MFA
            → Authentication completes successfully

    -> Example Scenario 2 (Code Change Required)
        -> Scenario: Application calls a middle-tier service. Middle-tier service calls a downstream API.
        -> Process:
            -> User signs in to the application
            -> Application sends token to middle-tier service
            -> Middle-tier performs On-Behalf-Of flow
            -> Requests access to downstream API
            -> Downstream API has Conditional Access policy
        -> Result
            -> Identity platform returns claims challenge
            -> Middle-tier sends challenge back to application
            -> Application must prompt user to complete additional authentication

    -> Key Concept (Claims Challenge)
        -> A claims challenge is returned when additional security verification is required.
        -> It informs the application that Conditional Access requirements must be satisfied.
        -> The application must handle the challenge and request additional authentication.

    -> Best Practice for Developers
        -> Implement Conditional Access challenge handling
        -> Use authentication libraries like MSAL
        -> Ensure applications can perform interactive sign-in when required
        -> Design applications to handle policy changes dynamically

    -> Final Summary
        -> Conditional Access is a security policy engine in Microsoft Entra ID.
        -> It enforces conditions such as MFA, device compliance, or location restrictions.
        -> Most applications work without modification.
        -> Some advanced scenarios require the application to handle Conditional Access challenges.
        -> Proper implementation ensures the application continues functioning when new security policies are applied.

=> Difference between Conditional Access vs RBAC vs Authentication vs Authorization
    1 → Authentication

    	→ Authentication means verifying the identity of a user or application.
    	→ It answers the question: "Who are you?"

    	Process
    	→ User provides credentials
    	→ Identity provider verifies credentials
    	→ User identity is confirmed

    	Examples
    	→ Username and Password
    	→ Multifactor Authentication (MFA)
    	→ Login using Microsoft Entra ID
    	→ OAuth login

    	Result
    	→ Identity is verified
    	→ Access token or ID token is issued

    --------------------------------------------------

    2 → Authorization

    	→ Authorization means determining what an authenticated user is allowed to do.
    	→ It answers the question: "What are you allowed to access?"

    	Process
    	→ User is authenticated
    	→ System checks permissions or policies
    	→ Access is granted or denied

    	Examples
    	→ Access to an API
    	→ Access to a database
    	→ Access to application features

    	Result
    	→ User receives permission to access specific resources

    --------------------------------------------------

    3 → Role-Based Access Control (RBAC)

    	→ RBAC is a type of authorization mechanism.
    	→ It controls access to Azure resources based on roles.

    	Concept
    	→ Instead of assigning permissions directly to users, roles are assigned.

    	Structure
    		User
    			→ Role Assignment
    				→ Resource

    	Examples
    	→ Owner
    	→ Contributor
    	→ Reader
    	→ Storage Blob Data Contributor

    	Purpose
    	→ Control access to Azure resources
    	→ Simplify permission management

    --------------------------------------------------

    4 → Conditional Access

    	→ Conditional Access is a security policy engine.
    	→ It decides whether authentication should be allowed based on conditions.

    	Concept
    	→ Access is allowed only when specific conditions are satisfied.

    	Examples of Conditions
    	→ Require Multifactor Authentication
    	→ Allow access only from corporate network
    	→ Allow access only from managed devices
    	→ Block access from risky locations

    	Purpose
    	→ Add extra security before granting access

    --------------------------------------------------

    5 → Relationship Between All Concepts

    	User tries to access application
    		→ Authentication verifies identity
    			→ Conditional Access checks security policies
    				→ Authorization determines permissions
    					→ RBAC enforces resource access

    --------------------------------------------------

    6 → Simple Real World Example

    	User tries to access Azure Blob Storage

    	Process
    	→ User signs in using Microsoft Entra ID (Authentication)
    	→ Conditional Access checks if MFA is required
    	→ If conditions satisfied, authentication continues
    	→ RBAC checks if user has Storage Blob Data Reader role
    	→ Authorization allows access to the storage resource

    --------------------------------------------------

    7 → Quick Comparison Table

    	Authentication
    	→ Verifies identity
    	→ Example: Login with username/password

    	Authorization
    	→ Determines what the user can access
    	→ Example: Permission to read files

    	RBAC
    	→ Role based authorization system in Azure
    	→ Example: Contributor role

    	Conditional Access
    	→ Security policy controlling how authentication happens
    	→ Example: Require MFA from outside corporate network

    --------------------------------------------------

    8 → Final Summary

    	Authentication
    	→ Who you are

    	Authorization
    	→ What you can access

    	RBAC
    	→ How Azure manages authorization

    	Conditional Access
    	→ Under what conditions access is allowed

=====================================================================================! PKCE !=====================================================================================

    => PKCE (PROOF KEY FOR CODE EXCHANGE) Definition:
        -> PKCE is a security extension to the Authorization Code Flow that protects against authorization code interception attacks.
        -> It is mainly used by Public Clients:
            -> Angular (SPA)
            -> Mobile apps
            -> Desktop apps

        -> WHY PKCE IS NEEDED:
            -> Public clients cannot store client_secret securely
            -> So attacker can intercept authorization code and exchange it for token
            -> PKCE solves this problem WITHOUT using client_secret

        -> HOW PKCE WORKS (STEP BY STEP):
            -> Step 1: Angular Generates Code Verifier
                -> A random string (very secure)
                -> Example:
                   code_verifier = "abc123xyz_random_secure_string"

            -> Step 2: Create Code Challenge
                -> Hash the verifier (usually SHA256)
                   code_challenge = HASH(code_verifier)

            -> Step 3: Send Request to Azure AD
                -> Angular sends code_challenge NOT code_verifier

            -> Step 4: User Logs In (Azure AD)
                -> Azure AD authenticates user
                -> Returns Authorization Code

            -> Step 5: Exchange Code for Token
                -> Angular sends:
                   - Authorization Code
                   - ORIGINAL code_verifier

            -> Step 6: Azure AD Verifies
                -> Azure AD hashes code_verifier again
                -> Compares with original code_challenge
                    IF MATCH:
                       -> Token is issued ✅
                    ELSE:
                       -> Request rejected ❌

        -> SECURITY BENEFIT:
            -> Even if attacker steals Authorization Code:
                -> They DO NOT have code_verifier
                -> So they cannot get access token

        -> FLOW SUMMARY:
            code_verifier (secret) → stays in frontend
            code_challenge → sent to Azure AD
        -> Azure AD verifies both → ensures request is genuine

        INTERVIEW ONE-LINER:
        --------------------
        "PKCE is a security mechanism used in Authorization Code Flow where a
        temporary secret (code_verifier) is used to securely exchange the authorization
        code for an access token, especially for public clients like Angular apps."

=====================================================================================! MSAL !=====================================================================================
=> Microsoft Authentication Library (MSAL)
    -> MSAL (Microsoft Authentication Library) is a library provided by Microsoft that helps your application:
        -> Authenticate users
        -> Get access tokens
        -> Call secure APIs like Microsoft Graph or your own backend
    -> Why MSAL is Important: Without MSAL:
        -> You must manually implement OAuth 2.0
        -> Handle token expiry
        -> Manage refresh tokens
    -> With MSAL:
        -> No need to deal with OAuth protocol directly
        -> Automatically handles token caching & refresh
        -> Supports multiple platforms
        -> Provides logging & debugging support
    -> MSAL doesn't authenticate directly. It request token from Microsoft Identity Platform
    -> Flow: User → MSAL → Microsoft Identity Platform → Token → App → API
    -> Supported Platforms: MSAL works everywhere:
        | Library          | Platform              |
        | ---------------- | --------------------- |
        | MSAL.NET         | .NET, MAUI, Xamarin   |
        | MSAL Java        | Java apps             |
        | MSAL.js          | JavaScript frameworks |
        | MSAL React       | React apps            |
        | MSAL Angular     | Angular apps          |
        | MSAL Python      | Python apps           |
        | MSAL Node        | Node.js               |
        | MSAL Android/iOS | Mobile apps           |

    -> So, You can use it in: Spring Boot (Java), React frontend, Microservices, Mobile apps
    -> Difference between Public vs Confidential client:
        | Feature        | Public Client | Confidential Client |
        | -------------- | ------------- | ------------------- |
        | Secret Storage | ❌ Not safe    | ✅ Safe              |
        | Runs on        | User device   | Server              |
        | Flow Type      | User-based    | App-based           |
        | Example        | React App     | Spring Boot API     |

    -> IDEAL AUTHENTICATION FLOW (AZURE AD + ANGULAR + SPRING BOOT)

        -> Step 1: User Opens Angular Application
            -> Angular app is a Public Client (cannot store secrets securely)
            -> MSAL library is already initialized
            -> Important: At this stage, NO PKCE values are generated yet

        -> Step 2: User Clicks Login
            -> Angular uses MSAL (Microsoft Authentication Library) login method. Example: msalService.loginRedirect()

        -> Step 3: MSAL Generates PKCE Values (INTERNALLY)
            -> MSAL automatically generates:
                -> code_verifier  (random secure string)
                -> code_challenge = HASH(code_verifier)
            -> Important:
                -> Developer does NOT write this logic manually
                -> MSAL handles it internally

        -> Step 4: Redirect to Azure AD
            -> MSAL sends request to Azure AD with: client_id, redirect_uri, scope, code_challenge
            -> Angular exchanges this code for Access Token (via MSAL)

        -> Step 5: User Authentication (Azure AD)
            -> User enters credentials
            ->  Azure AD validates identity

        -> Step 6: Authorization Code Returned
            ->  Azure AD redirects back to Angular with: Authorization Code

        -> Step 7: Token Exchange (PKCE Verification Happens Here)
            -> MSAL sends request to Azure AD with: Authorization Code & ORIGINAL code_verifier ✅
            -> Azure AD:
                -> Hashes code_verifier again
                -> Compares with earlier code_challenge
                IF MATCH:
                   -> Access Token issued ✅
                ELSE:
                   -> Request rejected ❌

        -> Step 8: Access Token Received
            -> Angular gets Access Token (with user claims)
            -> Token contains:
                -> User ID (sub)
                -> Username
                -> Roles / Permissions
                -> Expiry time

        -> Step 9: Angular Calls Spring Boot API
            -> Sends request with: Authorization: Bearer <Access_Token>
            -> Important:
                -> No need of user_key in header
                -> Token itself contains user information

        -> step 10: Spring Boot Backend (Resource Server)
            -> Validates token using Azure AD public keys
            -> Extracts user details from token claims

        -> Step 11: Authorization in Backend
            -> Based on:
                -> Roles
                -> Permissions
                -> Claims inside token

        -> Step 12: (Optional - Microservices Case)
        ---------------------------------------
        -> If API1 calls API2, API1 cannot directly forward same token came from public client because Token audience (aud) is meant for API1. API2 will reject it
            -> So, API1 Initiates OBO Flow
                -> API1 sends request to Azure AD with:
                    -> Its own client_id & client_secret
                    -> Incoming User Token
                    -> Requested scope for API2
                -> Azure AD Issues New Token
                    -> Returns a NEW Access Token
                    -> Important Point:
                        -> This token represents: "API1 calling API2 ON BEHALF OF USER"
                        -> Token contains: User identity & API2 as audience
                -> Now API1 Calls API2 with new token: Authorization: Bearer <OBO_Access_Token>
                -> API2 Validates Token: It will verify:
                    -> Token is valid
                    -> Audience = API2
                    -> User identity exists
                -> API2 Processes Request
                    -> Applies authorization based on: User roles & Permissions
           -> KEY POINTS:
              -----------
              -> Original token is NOT reused ❌
              -> New token is generated for downstream API ✅
              -> User context is preserved ✅
              -> Secure microservice communication ✅

              -> Think like this:
                  User → gives permission to API1
                  API1 → asks Azure AD: "Can I call API2 on behalf of this user?"
                  Azure AD → gives new token

        -> FLOW SUMMARY:

            Angular (MSAL)
               ↓
            Azure AD (Login + Token Issuance)
               ↓
            Access Token (contains USER identity)
               ↓
            Spring Boot API (Validates + Authorizes)

=> MSAL CLIENT APPLICATION INITIALIZATION
    -> Before using MSAL, we must initialize a client application
    -> MSAL provides builders to configure applications easily

    -> TWO TYPES OF CLIENT BUILDERS:
        -> PublicClientApplicationBuilder

                                .Build();

                    -> Client Credentials Flow

        -> .WithClientId()
            -> Override client ID if needed
        -> .WithTenantId()
            -> Specify tenant explicitly
        -> Logging & Debugging: Used for Troubleshooting authentication issues
            -> .WithLogging()
            -> .WithDebugLoggingCallback()

    -> STEP 4: CONFIDENTIAL CLIENT SPECIAL OPTIONS
        -> .WithClientSecret(string secret)
        -> .WithCertificate(X509Certificate2 cert)
        -> Important Point: You can use only one Client-secret or certificate. If both used → MSAL throws exception

================================================! SHARED ACCESS SIGNATURE (SAS) – AZURE STORAGE !======================

=> Definition: SAS (Shared Access Signature) is a secure URL that provides limited access to Azure Storage resources WITHOUT exposing account keys
    -> It is basically: URI + SAS Token
    -> Example: https://storageaccount.blob.core.windows.net/container/file.jpg?SAS_TOKEN
        -> https://medicalrecords.blob.core.windows.net/patient-images/patient-116139-nq8z7f.jpg?sp=r&st=2020-01-20T11:42:32Z&se=2020-01-20T19:42:32Z&spr=https&sv=2019-02-02&sr=b&sig=SrW1HZ5Nb6MbRzTbXCaPm%2BJiSEn15tC91Y4umMPwVZs%3D
        -> Split into:
        -> Resource URI: https://storageaccount.blob.core.windows.net/container/file.jpg
            -> https://medicalrecords.blob.core.windows.net/patient-images/patient-116139-nq8z7f.jpg?
        -> SAS Token: Contains permissions + expiry + signature
            ->  sp=r&st=2020-01-20T11:42:32Z&se=2020-01-20T19:42:32Z&spr=https&sv=2019-02-02&sr=b&sig=SrW1HZ5Nb6MbRzTbXCaPm%2BJiSEn15tC91Y4umMPwVZs%3D
    -> HOW SAS WORKS:
        -> Client uses SAS URL to access resource
        -> Azure Storage verifies:
            -> Signature (sig)
            -> Expiry time
            -> Permissions
        -> IF valid:
              -> Access granted ✅
           ELSE:
              -> Access denied ❌

    -> TYPES OF SAS:
        -> USER DELEGATION SAS (RECOMMENDED ✅)
            -> Secured using Azure AD (Microsoft Entra ID)
            -> No storage account key needed
            -> Used for:
                -> Blob Storage
                -> Data Lake
            -> BEST SECURITY ✅

        -> SERVICE SAS
            -> Secured using Storage Account Key
            -> Used for: Blob, Queue, Table & File Storage

        -> ACCOUNT SAS
            -> Secured using Storage Account Key
            -> Works across multiple services
            -> More powerful but less secure

    -> SAS TOKEN COMPONENTS:
        -> Example: sp=r&st=...&se=...&sv=...&sr=b&sig=...
            -> sp (Permissions)
                -> r: read
                -> w: write
                -> d: delete
                -> l: list
                -> Example:
                    sp=r  (read only)
                    sp=acdlrw (full access)

            -> st (Start Time): When access begins
            -> se (Expiry Time): After expiry → access automatically revoked
            -> sr (Resource Type): b = blob, c = container
            -> sig (Signature): Cryptographic signature. Generated using: Account key OR Azure AD credentials
                -> Used by Azure to validate request

        -> Scenario:
            -> Spring Boot backend generates SAS URL
            -> Angular frontend directly accesses Blob Storage

        -> Flow:
            Spring Boot
               ↓
            Generate SAS URL
               ↓
            Send to Angular
               ↓
            Angular uploads/downloads file directly from Azure Storage

    -> WHEN & HOW TO USE SAS (REAL-WORLD DESIGN PATTERNS)
        -> Use SAS when you want to give LIMITED, TEMPORARY access to storage to clients who do NOT have direct permissions
        -> SCENARIO 1: FRONT-END PROXY (TRADITIONAL APPROACH)
            -> Client sends request to backend
            -> Backend Authenticates user and Validates business logic
            -> Backend uploads/downloads file on behalf of client
            -> ADVANTAGES:
                -> Full control over: Validation, Security, Business rules

        -> SCENARIO 2: SAS PROVIDER (RECOMMENDED APPROACH ✅)
            -> Step 1: Client calls backend
            -> Step 2: Backend Authenticates user and Generates SAS URL
            -> Step 3: Client directly: Uploads / Downloads file from Azure Storage
            -> Advantage:
                -> High performance ✅
                -> Scalable ✅
                -> Reduced backend load ✅
            -> Disadvantage
                -> Limited control after SAS is issued ⚠️

        -> SCENARIO 3: HYBRID APPROACH (BEST PRACTICE 🔥)
            -> Flow
                -> Critical data → via Backend (validation required)
                -> Large files → via SAS (direct access)
            -> USE CASE:
                -> Validate metadata via backend
                -> Upload actual file via SAS

        -> SPECIAL CASE: SAS REQUIRED (COPY OPERATIONS)
            -> Blob → Blob (Different Storage Accounts): SAS REQUIRED for source ✅
            -> File → File (Different Storage Accounts): SAS REQUIRED for source ✅
            -> Optional: SAS can also be used for destination

        -> KEY DECISION RULE:
            IF:
            -> Small data + strict validation needed
               -> Use Backend Proxy

            IF:
            -> Large files / high traffic
               -> Use SAS ✅

    -> STORED ACCESS POLICY (SAS ADVANCED CONTROL)
        -> Stored Access Policy is a server-side configuration that controls SAS tokens
        -> It allows grouping and managing multiple SAS tokens centrally
        -> Works ONLY with Service SAS (not with User Delegation SAS)
        -> Problem with normal SAS:
            -> Once SAS is generated, it cannot be changed and Cannot revoke easily
        -> Solution:
            -> Stored Access Policy provides: Central control & Ability to modify/revoke SAS even after issuing ✅

        -> Supported Resource:
            -> Blob Containers
            -> File Shares
            -> Queues
            -> Tables

        -> HOW IT WORKS:
            -> Create Stored Access Policy on container/resource
            -> Generate SAS token referencing this policy (using policy ID)
            -> Client uses SAS token
            -> Important:
                -> SAS does NOT directly store permissions
                -> It refers to Stored Access Policy

        -> Structure:
            -> Stored Access Policy contains:
                -> Signed Identifier (policy name / ID)
                -> Permissions (r, w, d, l, etc.)
                -> Start Time
                -> Expiry Time
            -> SAS Token contains: Policy Identifier (instead of direct permissions)
            -> Key Rule: You cannot define same parameter in BOTH places
                -> Example: IF permissions defined in policy: Do NOT define in SAS token
            -> Advantage:
                -> Centralized Control : Manage multiple SAS tokens from one place
                -> Easy Revocatio: Revoke all SAS by modifying/deleting policy
                -> Dynamic Updates: Change permissions without regenerating SAS
            -> MODIFY / REVOKE POLICY:
                -> Modify Policy:
                    -> Change permissions / expiry time
                    -> Affects ALL associated SAS immediately
                -> Revoke Policy::
                    -> Option 1: Delete Policy
                    -> Option 2: Change expiry to past
                    -> Option 3: Change policy identifier (breaks link)
            -> Important Point:
                -> Policy changes may take ~30 seconds to reflect
                -> During this time: Requests may fail (403 Forbidden)

        -> Scenario: Backend generates SAS for multiple users
            -> Without Policy: Cannot revoke individual access easily
            -> With Policy: Assign SAS to a policy like:  "user-upload-policy"
            -> If needed: Disable uploads by modifying policy

        -> Flow:
            Spring Boot
               ↓
            Create Stored Access Policy
               ↓
            Generate SAS (linked to policy)
               ↓
            Send to Angular
               ↓
            Angular uses SAS

        -> Security & compliance data
    -> How it works:
        -> Step 1: Application authenticates using Azure AD (MSAL)
        -> Step 2: Receives Access Token
        -> Step 3: Calls Microsoft Graph API. Example: GET https://graph.microsoft.com/v1.0/users
        -> Step 4: Microsoft Graph validates token
        -> Step 5: Returns requested data
    -> What Data You Can Access: Using Microsoft Graph, you can access:
        -> User Info
        -> Emails (Outlook)
        -> Files (OneDrive / SharePoint)
        -> Teams messages
        -> Calendar events
        -> Devices
        -> Security data
    -> CORE COMPONENTS:
        -> MICROSOFT GRAPH API: This is main thing you use Rest API having single endpoint: https://graph.microsoft.com
            -> It is used in Angular, Spring Boot, Any Backend
        -> Microsoft Graph Connectors(Outside → Microsoft):
            -> It means bringing external data into Microsoft 365 so that Microsoft Search can show everything in one place
            -> Example: Data from Google Drive, Data from Jira, Data from Salesforce
        -> Microsoft Graph Data Connect(Microsoft → Azure Data Stores):
            -> It means exporting large data from Microsoft
            -> Example: Move email data → Azure Data Lake
            -> Use in analytics / AI
            -> Important: It is used for bulk data. It is more secure and scalable.

    -> Trick to remember:
        | Component    | Direction            | Purpose     |
        | ------------ | -------------------- | ----------- |
        | Graph API    | App ↔ Microsoft      | Fetch data  |
        | Connectors   | External → Microsoft | Import data |
        | Data Connect | Microsoft → Azure    | Export data |

    -> Intuition: Without Graph ---> Many APIs, With Graph ---> One API

    -> Query Microsoft Graph by using REST
        -> CORE CONCEPT:
            -> Microsoft Graph follows REST principles
            -> Uses HTTP methods to perform operations
            -> Uses JSON format for request and response

        -> BASE URL STRUCTURE:
            -> {HTTP Method} https://graph.microsoft.com/{version}/{resource}?{query-parameters}
            -> Example: GET https://graph.microsoft.com/v1.0/me

        -> REQUEST COMPONENTS:
            -> HTTP METHOD: Defines the action to perform on the resource. Methods Supported:
                GET     -> Read data
                POST    -> Create resource / perform action
                PATCH   -> Update partial resource
                PUT     -> Replace entire resource
                DELETE  -> Remove resource
                -> Important:
                    -> GET & DELETE → No request body required
                    -> POST, PATCH, PUT → Require JSON request body

            -> VERSION: Defines which version of API is used
                -> Available Version:
                    -> v1.0 (Stable / Production Ready): Should be used in production applications
                    -> beta (Preview / Experimental): May contain breaking changes.Use only for testing and development

            -> Resource: Represents the entity you want to access
                -> Example: me (current logged-in user), users, groups, messages

            -> RELATIONSHIPS: Resources can be connected
                -> Example: me/messages   (user emails),  me/drive   (user files)

            -> Actions: Some resource support actions. Example: me/sendMail
            -> QUERY PARAMETERS: Used to customize API response

        -> RESPONSE COMPONENTS:
            -> STATUS CODE: Indicates success or failure
                200 -> Success
                201 -> Created
                400 -> Bad Request
                401 -> Unauthorized
                403 -> Forbidden
                404 -> Not Found
            -> RESPONSE BODY: Contains requested data (JSON format). May be empty for some operations
            -> PAGINATION (@odata.nextLink):
                -> Used when response contains large data
                -> Microsoft Graph returns partial data
                -> Provides nextLink to fetch remaining data
                -> Example: @odata.nextLink: "https://graph.microsoft.com/..."

        -> NAMESPACE CONCEPT:
            -> Most Microsoft Graph APIs belong to: microsoft.graph
            -> Some APIs belong to sub-namespaces
            -> Example: microsoft.graph.callRecords
            -> Important: Unless specified, assume APIs are under microsoft.graph

        -> PERMISSIONS:
            -> Each API requires specific permissions (scopes). Example User.read, Mail.read, Files.read
            -> Important: Read operations need lower permissions but Write operations need higher permissions

        -> END-TO-END FLOW:
            -> Step 1: Register application in Azure AD
            -> Step 2: Authenticate user using MSAL
            -> Step 3: Obtain Access Token
            -> Step 4: Call Microsoft Graph API
            -> Step 5: Graph validates token & permissions
            -> Step 6: Returns response
