=> What is Docker: Docker is a containerization platform which allow you to package your application along with its dependency into a lightweight portable unit called a container.

=> What is a Dockerfile: A Dockerfile is a text file that contains instructions to build a Docker image.
    -> It tells Docker how to create my application image step-by-step.
    -> A Dockerfile is a set of instructions used by Docker to automatically build a Docker image.
    -> Why Do We Need Dockerfile?
        -> Without Dockerfile You manually install Java, copy jar file, set environment and run app.
        -> With docker everything is automated, Reproducible, Version Controlled, Same in Dev, test and Prod
        -> The Dockerfile is kept inside your application project folder, usually at the root level.
        -> Dockerfile itself is not executed like Java code. It is executed by Docker engine.
        -> When you run "docker build -t myapp ."
            -> Docker engine Reads Dockerfile ----> Executes instructions one by one ----> Creates image
        -> Complete Flow:
            -> Developer write the code ------> Adds Dockerfile in root -------> Pushes to Git
            -> CI/CD Pipeline pull code and run command to build image and push to docker hub or ACR(Azure Container Registery)
            -> Kubernetes pulls image from ACR and Runs container
            -> Flow:
                Application Code
                        ↓
                Dockerfile (inside project)
                        ↓
                docker build (runs locally or CI)
                        ↓
                Image Created
                        ↓
                Push to Registry (Docker Hub / ACR)
                        ↓
                Run as Container
    -> Basic Structure of a Dockerfile
        -> Typical Dockerfile contains:
            | Instruction | Purpose           |
            | ----------- | ----------------- |
            | FROM        | Base image        |
            | WORKDIR     | Working directory |
            | COPY / ADD  | Copy files        |
            | RUN         | Install packages  |
            | EXPOSE      | Application port  |
            | CMD         | Start application |
        -> Example Dockerfile
            # Use the .NET 6 runtime as base image
            FROM mcr.microsoft.com/dotnet/runtime:6.0

            # Set working directory
            WORKDIR /app

            # Copy application files
            COPY bin/Release/net6.0/publish/ .

            # Document application port
            EXPOSE 80

            # Run application
            CMD ["dotnet", "MyApp.dll"]

        -> Each Dockerfile instruction creates a new layer. Example:
            Base Image
               ↓
            Install Runtime
               ↓
            Copy Application
               ↓
            Configure Ports
               ↓
            Run Command

            These layers are cached.
            Benefits: Faster builds, Reusable layers, Smaller downloads

=> What is Image: An Image is a blueprint which contains application code, Runtime (Java, Node, Python etc.), Libraries, OS dependencies, Configuration.
    -> You create it using a dockerfile. Example: dockerfile
        FROM openjdk:17
        COPY app.jar app.jar
        ENTRYPOINT ["java","-jar","app.jar"]
    -> When you run docker build -t myapp . This creates an image which is read-only and doesn't run by itself.

=> What is container: A Container is a running instance of an image. Think like Image - class in java, Container - Object created from class
    -> In other words, A container is a lightweight package that contains:
        -> Application Code
        -> Runtime
        -> Libraries
        -> Dependencies
    -> All packed together so the application runs the same way everywhere.
    -> Containers are usually built using Docker.
    -> when you run "docker run myapp" Docker creates a container from the image and runs it.
    -> You can start, stop, Delete and scale containers. Example:
        Application Code
               ↓
        Dockerfile
               ↓
        Docker Image
               ↓
        Run as Container

=> What is Docker Hub: Docker Hub is a public container registry. It is like GitHub for Docker Images.

=> Azure Container Registry (ACR): Azure Container Registry (ACR) is Microsoft’s private Docker registry service inside Azure.
    -> It is similar to Docker Hub but private, secure, Integrated with Azure, Used in production environments
    -> Companies prefer ACR instead of Docker Hub because Private images, Enterprise security, Azure Kubernetes Service integration
    -> In real world when developers commit code, CI/CD tool(Azure Pipelines, Jenkins) build image and pushes to ACR.
    -> ACR Tasks allows Azure to build images for you instead of docker build, docker push. You can tell Azure “Whenever code changes, build image automatically.”
    -> ACR has 3 pricing levels:
        -> Basic: Learning, small projects
            -> Cheap, Limited Storage, Limited throughput, Good for Practice
        -> Standard: Most production apps
            -> More Storage, More speed, Used in Most real production setup
        -> Premium: High traffic, enterprise. Premium adds:
            -> Geo-replication: Store registry in multiple regions. Example: India, US, Europe. So users pull image from nearest region → Faster.
            -> Content Trust: Digitally sign images. Ensures image is not tampered.
            -> Private Link: Only allow private network access. No public internet access. Enterprise security feature.
    -> ACR support Linux images, Window images, OCI images, Helm charts. So it is not just Docker images — it supports OCI standard.
    -> Inside ACR each app has Repository name and Tags (versions) and each image is Read-only snapshot

    -> Step 2: You build an image: "docker build -t myapp"
    -> Step 3: You push image to a registry(Either push to Docker-Hub "docker push vinay/myapp" or ACR "docker push myacr.azurecr.io/myapp")

=> Encryption-at-Rest: When your image is stored in ACR, It is automatically encrypted. Even Microsoft cannot see raw image data. When someone pulls image → It decrypts automatically
    -> So, Push image → Azure encrypts → Stores safely and Pull image → Azure decrypts → Sends to you
    -> You don’t have to configure anything.
    -> For extra security option, You can use customer managed key(CMK):
        -> It means instead of Azure managing encryption key, You manage it in Azure Key Vault. This is for enterprise use case.

=> Regional Storage (Data Residency):
    -> When you create ACR in East US, Central India, West Europe. Azure stores registry data in that region.
    -> Region is Some companies have compliance rules like Data must stay in India, Data must stay in Europe.
    -> Paired Region Concept:
        -> In most regions (except Brazil South & Southeast Asia): Azure also keeps backup copy in paired region.
        -> Example: Central India ↔ South India (paired). But this does NOT mean automatic failover.
        -> If region fails: Registry may become unavailable, It does NOT automatically switch
        -> That’s why geo-replication exists.

=> Geo-Replication (Premium Feature):
    -> Available only in Premium tier. Geo-replication means:
        -> Your registry exists in multiple regions. Example:
            myacr
              ├── East US
              ├── West Europe
              └── Central India
        -> Benifit: High availability, Faster image pull, Disaster protection
        -> If India region fails → Europe still works.

=> Zone Redundancy
    -> It is also a premium features.
    -> Inside one region, Azure has multiple availability zones.
    -> Zone redundancy means: Your registry is replicated across 3 zones in same region.
    -> So if one data center fails, Registry still works.
    -> It protects against datacenter-level failure, not region-level.

=> What Happens During Regional Outage?
    -> Region down → Registry unavailable
    -> With geo-replication:
        -> Region A down → Region B continues working
    -> So for production, Premium + geo-replication is recommended.
    -> But sometimes, government regulations say: “Data must stay inside the country”
        -> In this case cross-country geo-replication is not allowed. But you still have multiple high-availability options with ACR(Azure Container Registery)
            -> Same-Country Geo Replication
                -> If Azure has multiple regions inside the same country, you can replicate there.
                -> Example: Central India, South India, East India
                -> Architecture:
                    ACR
                     ├── Central India
                     └── South India
            -> Zone Redundancy (Most Common for Sensitive Data)
                -> Instead of multiple regions, use Availability Zones inside the same region. Example:
                    Central India
                       ├── Zone 1
                       ├── Zone 2
                       └── Zone 3
                -> This protects against Datacenter failure, Hardware failure, Power failure
            -> Separate Registries per Country
                -> Large enterprises often do this. Example:
                    India Registry → myacr-india.azurecr.io
                    US Registry → myacr-us.azurecr.io
                    Europe Registry → myacr-eu.azurecr.io
                -> Each region manages its own images.
    -> Real Government / Banking Architecture
        -> For regulated industries: Banking, Government, Healthcare. Typical setup:
            ACR (Central India)
               ├── Zone redundant
               └── Private endpoint
        -> No geo-replication outside country.

=> Important Thing About Container Images
    -> Container images usually contain: Application code, Runtime, Dependencies
    -> They normally don't contain sensitive user data.
    -> Sensitive data is usually stored in:
        -> Azure SQL Database
        -> Azure Cosmos DB
        -> Azure Key Vault
    -> So sometimes companies allow geo-replication for images because images themselves are not confidential.
    -> But compliance rules vary.
    -> What if data cannot leave the country?
        -> In such scenarios geo-replication across countries is not allowed due to data residency requirements.
        -> Instead we can configure geo-replication only across regions within the same country or use zone redundancy within a single region to ensure high availability while keeping data inside the country.

=> How AKS(Azure Kubernetes Service) pulls images from ACR(Azure Container Registry) securely without storing username/password.
    -> Suppose your container image is stored in ACR: myacr.azurecr.io/myapp:v1
    -> Your Kubernetes cluster (AKS) needs to pull this image to run the container.
    -> Normally in Docker you would do: docker login myacr.azurecr.io, This requires username and password, But storing credentials in Kubernetes is not secure.
    -> So Azure uses Identity-based authentication instead.
    -> Azure uses Managed Identity to authenticate automatically. Flow:
        AKS Cluster
             │
             │ Managed Identity
             │
             ▼
        Azure Active Directory
             │
             ▼
        ACR authorization
    -> So AKS does not store passwords. Internally:
        -> When AKS pulls image
            -> AKS cluster has Managed Identity
            -> Azure verifies identity in Azure AD
            -> Azure checks permissions on ACR
            -> If allowed → image is pulled

    -> Earlier people used Kubernetes secrets. But this method has problems:
        -> ❌ Credentials stored in cluster
        -> ❌ Security risk
        -> ❌ Hard to rotate passwords
        -> So Azure recommends Managed Identity.

=> Azure Container Instance:
    -> In modern cloud applications, developers package their applications and dependencies into containers. Containers make applications portable, lightweight, and easy to deploy across different environments.
    -> To run containers in the cloud, cloud providers offer services such as Azure Container Instances, which allow developers to run containers without managing infrastructure.
    -> Azure Container Instances is a serverless container service that allows you to run containers directly in Azure without creating or managing virtual machines.
    -> In simple terms:
        -> Traditional approach: Application → VM → OS → Infrastructure
        -> ACI approach: Application → Container → Azure manages everything
        -> So developers only focus on application code, while Azure manages the infrastructure.

    -> Definition:
        -> Azure Container Instances (ACI) is a cloud service that lets you quickly run Docker containers in Azure without managing servers or orchestration tools like Kubernetes.
        -> It is the fastest and simplest way to run containers in Azure.
        -> Key idea: Write application → create container image → run in ACI
        -> Azure automatically handles: Virtual machines, Networking, Container runtime, Scaling infrastructure

    -> Why Azure Container Instances is Needed
        -> Before ACI, developers had two options: Virtual Machines & Kubernetes
            -> Virtual Machines: You had to create VM, install Docker, manage OS updates, configure networking. This required infrastructure management.
            -> Kubernetes: Services like Azure Kubernetes Service provide orchestration but are more complex.
                -> ACI solves this problem by providing simple container execution.
                -> ACI allows you to run containers in seconds without VM provisioning.

    -> Key Features of Azure Container Instances
        -> Fast Startup
        -> No Infrastructure Management
        -> Serverless Pricing
        -> Supports Public Container Registries: ACI can pull images from multiple container registries such as:
            -> Docker Hub
            -> Microsoft Container Registry
            -> Azure Container Registry
            -> This allows easy deployment of container images
        -> Container Groups in ACI: ACI introduces a concept called Container Group.
            -> A container group is a group of one or more containers that run together on the same host and share resources.
            -> They share network, storage, lifecycle
            -> This is similar to a Pod in Kubernetes.
            -> Example:
                Container Group
                   ├── Web Application Container
                   ├── Logging Container
                   └── Monitoring Container
            -> All containers run together and communicate internally.
        -> Networking in ACI: Azure Container Instances allows containers to be exposed to the internet.
            -> When you create a container instance, Azure can assign: Public IP address, DNS name
            -> Users can access the application using this DNS endpoint.
        -> Persistent Storage:
            -> Containers are normally stateless, meaning data disappears when the container stops. Example:
                Container starts
                   ↓
                Application creates file /data/report.txt
                   ↓
                Container stops
                   ↓
                File disappears ❌
            -> This happens because containers store data in temporary container storage. So containers are called stateless.
            -> Containers Are Designed This Way because container are meant to be lightweight, replaceable, scalable
            -> Cloud systems often destroy and recreate containers automatically.
            -> Azure solves this problem by providing external storage for containers. In Azure Container Instances, you can mount Azure Files.
            -> Think of it like attaching a cloud hard drive to the container.
            -> With Azure File Share Instead of saving data inside the container, the container writes data to Azure storage.
            -> Example: Azure Files → mounted to container. This allows data persistence even if the container stops.
            -> Architecture:
                -> Without external storage:
                    Container
                       └── data.txt

                    Container stops
                       ❌ data lost
                -> With Azure File Share:
                    Container
                       └── mounted storage
                             ↓
                          Azure File Share
                             └── data.txt

                    Container stops
                       ✅ data still exists
        -> Security in Azure Container Instances:
            -> ACI provides hypervisor-level isolation, which means containers run securely and are isolated like virtual machines.
            -> Security features include:
                -> secure TLS connections
                -> container isolation
                -> managed identity integration
            -> Managed identity allows containers to securely access Azure services without storing credentials.
        -> Linux and Windows Container Support
            -> ACI supports both Linux and window container
            -> You can select the operating system while creating the container instance
        -> Integration with Virtual Networks
            -> ACI can run containers inside an Azure Virtual Network (VNet).
            -> Benefits: secure communication with databases, connection with on-premise systems, internal microservice communication
        -> Integration with Kubernetes
            -> ACI can integrate with Azure Kubernetes Service.
            -> This enables elastic bursting.
            -> Example: Normal workload → runs in AKS, Traffic spike → extra containers run in ACI
            -> This allows automatic scaling without adding more servers.
        -> Common Use Cases of Azure Container Instances
            -> Batch processing jobs: data transformation, report generation
            -> Build tasks: CI/CD pipeline container builds
            -> Event driven processing: queue triggered processing
            -> Quick application testing: Developers can quickly test containers without full infrastructure.
        -> Limitations of ACI
            -> Although ACI is powerful, it has some limitations. ACI is not designed for large microservice architectures.
            -> For complex applications requiring: auto scaling, service discovery, rolling deployments
            -> Microsoft recommends Azure Kubernetes Service.

    -> Restart Policy:
        -> Why Restart Policies Are Needed:
            -> Many containers in the cloud are used to perform temporary tasks like: build an application, run tests, process images, run scripts, batch data processing.
            -> Once the script finishes, the container does not need to run anymore.
            -> However, sometimes you may want container to restart automatically, run only once.
            -> To control this behavior, Azure provides restart policies.
        -> Definition: A restart policy defines What Azure should do with a container after it stops.
            -> In Azure Container Instances, when you create a container group, you can choose how the container should behave after it exits.
            -> There are three restart policies:
                -> Always(Default): Container restarts automatically every time it stops
                -> Never: Container runs only once and never restarts
                -> OnFailure: Container restarts only if it fails
        -> Run to Completion Concept: Some containers are designed to finish their work and exit. Fpr example: Process 10k images
            -> Steps: Container starts--->Images processed--->Application exits--->Container stops
            -> This is called Run to completion. These workloads typically use: Restart Policy = Never or Restart Policy = OnFailure
        -> Container Status After Completion: When a container finishes its work, Azure changes its status to: Terminated
            -> It means Container finished execution
            -> This happens when restart policy is: Never, OnFailure (and task succeeded)
        -> Flow:
                       Container stops
                              │
                  ┌───────────┼───────────┐
                  │           │           │
               Always       Never     OnFailure
                  │           │           │
            Restart        Stop       Restart if
            always       permanently     failed

    -> Container deployment configuration:
        -> Below YAML file defines a container deployment configuration for Azure Container Instances.
        -> It creates a container that runs an application and stores logs in an Azure file share so the data persists even after the container stops.
        -> This configuration will:
            -> Create a container group
            -> Run a container image
            -> Expose it on port 80 (public internet)
            -> Mount an Azure file share for persistent storage
            -> Allow access using a DNS name
            -> Architecture:
                Internet User
                      ↓
                Public DNS (aci-demo)
                      ↓
                Azure Container Instance
                      ↓
                Container (hellofiles)
                      ↓
                Mounted Azure File Share
            -> YAML Configuration file:
                apiVersion: '2019-12-01'  # API version used to create resources in Azure Container Instances
                location: eastus  # Azure region where the container instance will be deployed
                name: file-share-demo  # Name of the container group
                properties:  # Main configuration section for the container group
                  containers:  # List of containers inside this container group
                  - name: hellofiles  # Name of the container
                    properties:  # Container specific configuration
                      environmentVariables: []  # No environment variables are passed to the container
                      image: mcr.microsoft.com/azuredocs/aci-hellofiles  # Container image stored in Microsoft Container Registry
                      ports:  # Ports exposed by the container
                      - port: 80  # Container listens on port 80 (typically used for HTTP web servers)
                      resources:  # Resource allocation for the container
                        requests:
                          cpu: 1.0  # Allocate 1 CPU core
                          memoryInGB: 1.5  # Allocate 1.5 GB RAM
                      volumeMounts:  # Mount external storage inside the container
                      - mountPath: /aci/logs/  # Directory inside container where storage will be mounted
                        name: filesharevolume  # Name of the volume defined later in YAML
                  osType: Linux  # Operating system used by the container
                  restartPolicy: Always  # Container automatically restarts if it stops
                  ipAddress:  # Network configuration for accessing the container
                    type: Public  # Container will have a public IP address accessible from the internet
                    ports:
                      - port: 80  # Expose port 80 publicly so users can access the application
                    dnsNameLabel: aci-demo  # DNS prefix that Azure will use to generate public URL
                  volumes:  # Define storage volumes available to containers
                  - name: filesharevolume  # Volume name referenced earlier in volumeMounts
                    azureFile:  # Volume is backed by Azure File Share storage
                      shareName: acishare  # Name of the Azure file share
                      storageAccountName: <Storage account name>  # Azure Storage account containing the file share
                      storageAccountKey: <Storage account key>  # Authentication key for accessing the storage account
                tags: {}  # Optional metadata tags (empty in this example)
                type: Microsoft.ContainerInstance/containerGroups  # Resource type being created (ACI container group)

=> Azure Container Apps:
    -> Azure Container Apps is a serverless container platform built on top of Azure Kubernetes Service.
    -> This means you can run containers without managing Kubernetes clusters and Azure automatically manages infrastructure, scaling, and networking.
    -> Architecture Flow:
        Developer
           ↓
        Container Image
           ↓
        Azure Container Apps
           ↓
        Runs on Azure Kubernetes Service
           ↓
        Auto scaling + networking + monitoring
    -> Microsoft designed Azure Container Apps mainly for microservices and event-driven applications.
    -> Container Apps automatically scale based on different triggers. Scaling engine is powered by KEDA (Kubernetes Event Driven Autoscaling).
    -> Container Apps support multiple versions of an application.
        Version 1 → current production
        Version 2 → new release
        -> You can run both versions and control traffic. This enables:
            | Deployment Strategy   | Meaning                             |
            | --------------------- | ----------------------------------- |

            | A/B Testing           | send partial traffic to new version |
        -> HTTPS Ingress
            -> Ingress means how users access the container app from the internet.
            -> Azure Container Apps provides: automatic HTTPS, secure endpoint, no load balancer setup required
            -> Example: https://myapp.azurecontainerapps.io. Azure automatically manages certificates.
        -> Internal Service Communication
            -> Container Apps also support internal networking.
            -> This allows microservices to communicate securely.
            -> Example architecture:

                     ↓
                Order Service
                     ↓
                Payment Service
            -> These services communicate internally using: internal ingress, DNS based service discovery

            -> Container Apps integrate with Dapr(Distributed Application Runtime).

            -> Container Registries Supported

=> What is Dapr: Dapr (Distributed Application Runtime) is:
    -> An open-source project under the CNCF (Linux Foundation).
    -> A set of incrementally adoptable features for building distributed microservices.
    -> Designed to simplify communication between services, state management, pub/sub, secrets, and more
    -> Azure Container Apps (ACA) provides:
        -> Managed Dapr integration – no need to deploy Dapr yourself.
        -> Automatic upgrades – Dapr versions are handled by Azure.
        -> Simplified API – you interact with Dapr easily via HTTP/gRPC calls.
        -> Think of it as: your container apps get a built-in Dapr “sidecar” that handles service-to-service communication, messaging, and more.
        -> Core Dapr APIs in ACA
            | API                               | Description                                                      |
            | --------------------------------- | ---------------------------------------------------------------- |
            | **Service-to-service invocation** | Call other container apps reliably with **mTLS encryption**.     |
            | **State management**              | Store and retrieve state for CRUD operations and transactions.   |
            | **Pub/sub**                       | Send messages via a broker between publishers and subscribers.   |
            | **Bindings**                      | Trigger your app from external events (e.g., queue, timer).      |
            | **Actors**                        | Single-threaded, message-driven units of work that scale easily. |
            | **Observability**                 | Send traces to **Application Insights**.                         |
            | **Secrets**                       | Access secure values from your app or secret stores.             |
            | **Configuration**                 | Retrieve or subscribe to config items in supported stores.       |
        -> How Dapr works in ACA
            -> Dapr-enabled container app: Enabled at container app level, applies to all revisions.
            -> Dapr sidecar: Runs alongside your container, exposes APIs at:
                -> HTTP: port 3500, gRPC: port 50001
            -> Components: Pluggable modules providing functionality like pub/sub, state store, secret store.
                -> Components can be shared across apps or scoped to specific apps.
                -> Scope ensures only intended apps use a component.
        -> You can enable Dapr in ACA via: Azure CLI, Infrastructure as Code – Bicep or ARM templates, Azure Portal
